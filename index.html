CORS

CORS即CROSS ORIGIN REQUEST SHARE是一种浏览器请求机制。

是一种基于HTTP头的机制。

为了保障跨源请求的安全

现代浏览器有两种请求API，XMLHttpRequest和Fetch API，均对请求的同源性有要求，如果请求的是跨源资源，则需要响应报文包含了正确的CORS响应头。

什么情况下需要CORS？

详细cross-origin sharing standard

访问控制场景

1. 简单请求

如果满足以下条件，则可视为简单请求：

摘自MDN:

  • 使用下列方法之一：
    • GET
    • HEAD
    • POST
  • 除了被用户代理自动设置的首部字段（例如 Connection，User-Agent）和在 Fetch 规范中定义为 禁用首部名称 的其他首部，允许人为设置的字段为 Fetch 规范定义的 对 CORS 安全的首部字段集合。该集合为：
    • Accept
    • Accept-Language
    • Content-Language
    • Content-Type （需要注意额外的限制）
    • Content-Type 的值仅限于下列三者之一：
    • text/plain
    • multipart/form-data
    • application/x-www-form-urlencoded
  • 请求中的任意 XMLHttpRequest 对象均没有注册任何事件监听器；XMLHttpRequest 对象可以使用 XMLHttpRequest.upload 属性访问。
  • 请求中没有使用 ReadableStream 对象。

当使用简单请求时，仅需要使用请求头的Origin和Access-Control-Allow-Origin就能完成最简单的访问控制。

2. 预检请求

当不进行简单请求时，在进行跨源请求时，需要先使用OPTIONS方法发起一个预检请求到服务器，来判断服务器是否允许此请求。

可以避免跨域请求对服务器的用户数据产生未预期的影响。

3. 附带身份凭证的请求

进行跨域请求时，如果不进行特殊处理，不会发送身份凭证信息，如果需要发送凭证，在请求时需要设置withCredentials为true，响应头Access-Control-Allow-Credentials也必须要设置为true，请求才能正确的完成。

在进行附带身份凭证的请求时：

  • 服务器不能将Access-Control-Allow-Origin设置为*
  • 服务器不能将Access-Control-Allow-Headers设置为*
  • 服务器不能将Access-Control-Allow-Methods设置为*

引入cdn的一些小发现

最近在用unpkg、jsdelivr的cdn服务时发现，自动生成标签有一个SRI的选项，勾选后就会在<script>标签中添加两个新属性，举个栗子

<script src="https://cdn.jsdelivr.net/npm/jquery@3.6.0/dist/jquery.min.js" integrity="sha256-/xUj+3OJU5yExlq6GSYGSHk7tPXikynS7ogEvDej/m4=" crossorigin="anonymous"></script>

使用了两个属性：integrity和crossorigin，分别看一下

  • integrity

    SRI全称Subresource Integrity，即子资源完整性，用于浏览器检查从CDN处获取的资源是否被篡改的一个特性，通过验证哈希值来判断是否被篡改。

    将文件进行哈希，得到的哈希值进行BASE64转换后和哈希生成算法名一起写入integrity属性，格式为${SHA_CALC}-${SHA_BASE64_Result}，SHA_CALC目前只支持sha256、sha384、sha512

    可以使用openssl自行运算：

    openssl dgst -sha384 -binary FILENAME.js | openssl base64 -A

  • crossorigin

    crossorigin为部分html元素提供了CORS请求支持。

    CORS请求

    • <audio>
    • <img>
    • <link>
    • <script>
    • <video>

    默认情况下，即crossorigin属性不被设置时，将不会发起CORS请求。

    在非同源情况下，设置crossorigin为anonymous将不会通过 cookies，客户端 SSL 证书或 HTTP 认证交换用户凭据。

    在请求需要用户凭据的资源时，即使是同源，也需要将crossorigin设置为use-credentials才能正确的进行请求。
